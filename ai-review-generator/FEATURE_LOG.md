### Architecture Decisions

#### Why Ollama-first?
Ollama runs locally — no API costs, no internet dependency, no data leaving the device. Ideal for a shop environment where the server runs on a local machine. Cloud LLMs are available as optional fallbacks for when Ollama is not running.

#### Why no build step for the frontend?
The frontend is served by the same Express server and used on a single device (the shop's tablet/phone). A build step adds complexity with no benefit. ES6 modules are natively supported in all modern browsers.

#### Why Jaccard bigrams for deduplication?
Jaccard similarity on word bigrams is:
- Dependency-free (pure JS, works in both Node.js and browser)
- Fast enough for short review texts (O(n) per comparison)
- Language-agnostic (works for both English and Marathi)
- Tunable via `config.history.similarityThreshold`

Alternatives considered: cosine similarity on TF-IDF vectors (too heavy for browser), Levenshtein distance (too strict for paraphrased text), exact match (too lenient).

#### Why Express over a serverless/edge approach?
The system runs on a local machine in the shop. Express is simple, well-understood, and requires no cloud infrastructure. The server also acts as a security boundary — API keys never reach the browser.

#### Why flat JSON files for fallback pools?
No database is needed. The fallback pools are small (10–20 reviews per file), rarely change, and are read-only at runtime. Flat JSON files are easy to edit, version-control, and understand.
